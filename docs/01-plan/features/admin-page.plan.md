### Value Delivered

| 관점 | 설명 |
|------|------|
| **Problem** | 현재 모든 데이터(설문, 응원, 의견, 참관인, 소식)를 Supabase 대시보드에서만 확인/관리할 수 있어, 비개발자인 후보가 직접 운영하기 어렵다. |
| **Solution** | 비밀번호 보호된 관리자 페이지를 구축하여, 접수된 데이터를 한눈에 보고, 활동 소식을 직접 작성/관리할 수 있게 한다. |
| **Function UX Effect** | 후보(관리자)가 Supabase 없이도 스마트폰에서 접수 현황을 확인하고, 소식을 바로 게시할 수 있다. |
| **Core Value** | 후보가 사이트를 자립적으로 운영할 수 있게 하여, 유권자와의 소통을 즉각적으로 관리할 수 있다. |

## 1. 배경 및 목적

### 1.1 현재 상태
- **데이터 확인**: Supabase 대시보드 로그인 필요 (개발자 도구)
- **소식 작성**: Supabase Table Editor에서 직접 SQL/Row 삽입 필요
- **알림**: 이메일 알림으로 접수 사실만 확인 가능, 상세 목록 불가

### 1.2 문제점
1. 비개발자가 Supabase 대시보드를 사용하기 어려움
2. 소식 작성을 위해 매번 개발자 도움이 필요
3. 접수 데이터의 전체 현황을 파악하기 어려움
4. 모바일에서 Supabase 대시보드 사용이 불편함

### 1.3 목표
- 간단한 비밀번호 인증으로 보호된 `/admin` 페이지 구축
- 5개 테이블 데이터를 목록으로 조회
- 활동 소식(posts) CRUD 기능
- 모바일 친화적 UI

## 2. 기능 정의

### 2.1 A1: 관리자 인증 (우선순위: P0)
- 단순 비밀번호 인증 (환경변수 `ADMIN_PASSWORD`와 비교)
- 별도 회원가입/DB 불필요
- 세션 유지: 브라우저 sessionStorage
- 로그아웃 버튼

### 2.2 A2: 대시보드 (우선순위: P0)
- 각 테이블의 총 건수 카드로 표시
  - 설문 응답 수
  - 응원 메시지 수
  - 주민 의견 수
  - 참관인 신청 수
  - 소식 게시 수
- 최근 접수 건 3개 미리보기

### 2.3 A3: 데이터 목록 조회 (우선순위: P0)
- **설문 응답** (`survey_responses`): 위치, 불편유형, 상세, 연락처, 날짜
- **응원 메시지** (`cheers`): 이름, 메시지, 날짜
- **주민 의견** (`opinions`): 카테고리, 제목, 내용, 연락처, 날짜
- **참관인 신청** (`poll_observers`): 이름, 연락처, 유형, 가능시간, 날짜
- 각 목록: 최신순 정렬, 페이지네이션 (20건씩)

### 2.4 A4: 활동 소식 관리 (우선순위: P0)
- **작성**: 제목, 본문, 이미지 URL 입력
- **수정**: 기존 소식 내용 편집
- **삭제**: 확인 후 삭제
- 목록에서 바로 작성/수정/삭제

### 2.5 A5: 데이터 삭제 (우선순위: P1)
- 스팸성 응원/의견 개별 삭제
- 삭제 전 확인 다이얼로그

## 3. 기술 설계

### 3.1 인증 방식
- API Route (`/api/admin/login`)에서 비밀번호 검증
- 환경변수: `ADMIN_PASSWORD`
- 성공 시 서버에서 간단한 토큰 반환, sessionStorage에 저장
- 모든 admin API에 토큰 검증 미들웨어

### 3.2 데이터 접근
- Admin API Routes에서 `SUPABASE_SERVICE_ROLE_KEY` 사용 (RLS 우회)
- 프론트엔드 → Admin API → Supabase (service_role)

### 3.3 라우트 구조
```
/admin              → 로그인 또는 대시보드
/admin/surveys      → 설문 응답 목록
/admin/cheers       → 응원 메시지 목록
/admin/opinions     → 주민 의견 목록
/admin/observers    → 참관인 신청 목록
/admin/posts        → 소식 관리 (CRUD)
```
